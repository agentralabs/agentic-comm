\begin{abstract}
Multi-agent AI systems require structured communication infrastructure---typed messages, delivery guarantees, persistent history, and identity-aware routing---yet current frameworks treat inter-agent communication as untyped string passing through ephemeral function calls.
We present \textsc{AgenticComm}, a communication library that stores agent interactions in a compact binary format (\texttt{.acomm}) with eight message types, four channel topologies, three delivery guarantee levels, and cryptographic agent identity integration.
The key insight is that agent communication is not a transport problem but a \emph{structured artifact} problem: conversations are typed, channels have topology, delivery has contracts, and the full history must survive context window resets.
\textsc{AgenticComm} achieves sub-microsecond trust lookups (33.2\,ns at 10K agents), message send latency of 0.89\,$\mu$s per message, file persistence of 10K messages in 9.71\,ms, and storage efficiency of 71.8 bytes per message under Zstd compression---a 2.9:1 ratio over uncompressed bincode.
The system supports pub/sub topic matching with MQTT-style wildcards (183\,$\mu$s for 10K subscriptions), ChaCha20-Poly1305 and AES-256-GCM encryption (895\,ns and 949\,ns respectively), and end-to-end write pipelines including trust validation, consent checking, and persistence at 21.4\,ms for 10K messages.
\textsc{AgenticComm} is implemented in Rust, exposed through MCP (Model Context Protocol) via 17 consolidated tools, and designed as a portable, dependency-free communication substrate for long-running agent systems.
\end{abstract}

\section{Introduction}
\label{sec:intro}

The emergence of multi-agent AI systems---where multiple language model instances collaborate on complex tasks---has created a need for communication infrastructure that goes beyond simple function calls and string passing.
Current agent frameworks (LangChain~\cite{langchain2023}, CrewAI~\cite{crewai2024}, AutoGen~\cite{autogen2023}, MetaGPT~\cite{hong2023metagpt}) provide task orchestration but treat inter-agent communication as an implementation detail, typically passing untyped strings between agent functions.

This ad-hoc approach creates five problems that become acute as agent systems scale beyond simple pipelines:
(1)~\textbf{No structure}: messages have no type, schema, or expected response pattern---a command looks identical to a notification;
(2)~\textbf{No history}: communication is ephemeral and vanishes when the context window resets;
(3)~\textbf{No routing}: every connection is point-to-point, requiring modification of all agents when adding a new participant;
(4)~\textbf{No contracts}: there are no delivery guarantees, timeout policies, or acknowledgment protocols;
(5)~\textbf{No identity}: messages carry no cryptographic provenance and cannot be trust-gated.

The key insight behind \textsc{AgenticComm} is that agent communication is not a transport problem---it is a \emph{structured artifact} problem.
Conversations have types.
Channels have topology.
Delivery has contracts.
History must be navigable, searchable, and portable---surviving context window resets, process restarts, and even provider switches.
The \texttt{.acomm} file is the communication substrate: a single portable binary artifact containing the complete structured history of agent interactions.

\begin{itemize}[nosep, leftmargin=*]

\section{Background and Related Work}
\label{sec:related}

\subsection{Message-Oriented Middleware}

Traditional message brokers---RabbitMQ~\cite{rabbitmq2012}, Apache Kafka~\cite{kafka2011}, ZeroMQ~\cite{zeromq2013}, NATS~\cite{nats2021}---provide robust messaging for distributed systems but require running servers, operational expertise, and network infrastructure.
They do not integrate with agent identity, cognitive models, or LLM tool-calling protocols.
\textsc{AgenticComm} is a library, not a server: the \texttt{.acomm} file is the entire communication substrate.

\subsection{Agent Communication Languages}

FIPA-ACL~\cite{fipa2002} and KQML~\cite{kqml1994} defined formal agent communication languages with performatives (message types) and interaction protocols.
These were designed for symbolic AI agents in the 1990s---not for language model agents with large message content, context window constraints, and tool-calling interfaces.
\textsc{AgenticComm} draws on the performative concept (our message types are analogous to FIPA performatives) but targets modern LLM-based agents.

\subsection{Multi-Agent Frameworks}

LangChain~\cite{langchain2023} provides chain composition.
CrewAI~\cite{crewai2024} adds role-based agent teams.
AutoGen~\cite{autogen2023} supports multi-agent conversation.
MetaGPT~\cite{hong2023metagpt} introduces standardized operating procedures.
CAMEL~\cite{li2023camel} explores communicative agents.
These frameworks focus on task orchestration, treating communication as string passing.
None provide persistent history, typed messages, pub/sub routing, delivery guarantees, or cryptographic identity integration.

\subsection{Model Context Protocol}

MCP~\cite{mcp2024} defines a JSON-RPC protocol for connecting AI models to external tools.
\textsc{AgenticComm} uses MCP as its primary interface, exposing 17 consolidated tools that cover channel management, message operations, queries, trust, encryption, and session lifecycle.

\section{Architecture}
\label{sec:architecture}

\textsc{AgenticComm} consists of four layers: (1)~\textbf{CommStore}---persistence layer managing \texttt{.acomm} files with bincode serialization and Zstd compression; (2)~\textbf{Channel Manager}---lifecycle management for four channel types with configurable delivery semantics; (3)~\textbf{MessageEngine}---unified processing pipeline for trust validation, consent checking, routing, and query dispatch; (4)~\textbf{MCP Interface}---tool exposure layer for AI assistant integration.

\subsection{Message Types}
\label{sec:messages}

We define eight message types inspired by FIPA performatives~\cite{fipa2002} but adapted for LLM agent workflows.
Each type has a defined response expectation that enables protocol enforcement.

\textbf{Pub/Sub Topic Matching.}
Topics follow a dot-separated hierarchy with three matching modes: \emph{exact} (\texttt{build.frontend.done} matches only itself), \emph{single-level wildcard} (\texttt{build.*.done} matches \texttt{build.frontend.done} and \texttt{build.backend.done}), and \emph{multi-level wildcard} (\texttt{build.\#} matches \texttt{build}, \texttt{build.frontend}, and \texttt{build.frontend.done}).
Our benchmarks show topic matching across 10K subscriptions completes in 183\,$\mu$s (Section~\ref{sec:evaluation}).

\subsection{Binary File Format}
\label{sec:format}

The \texttt{.acomm} binary format stores communication history in a compact, checksummed container.
Figure~\ref{fig:format} shows the layout.

\label{fig:format}
\end{figure}

The header (96 bytes) contains magic bytes (\texttt{ACOMM001}), a format version, feature flags (currently: \texttt{FLAG\_ZSTD} for compression mode), entity counts, and timestamps.
Messages are serialized with bincode~\cite{bincode2023} and compressed with Zstd~\cite{zstd2016} at level~3, achieving 71.8 bytes per message at scale (Section~\ref{sec:storage}).
The footer contains a Blake3~\cite{blake3_2020} cryptographic hash covering all preceding sections.

\textbf{Atomic Writes.}
Writes use an atomic protocol: serialize to a temporary file, compute the Blake3 checksum, write the footer, then atomically rename.
This ensures readers never see partially-written data.
Interrupted writes leave the previous valid snapshot intact.

\textbf{Backward-Compatible Compression.}
The \texttt{FLAG\_ZSTD} header bit enables transparent migration: files written with the previous gzip compressor (flate2) lack this flag, so the reader falls back to gzip decompression.
New writes always use Zstd.

\subsection{Query Model}
\label{sec:queries}

\textsc{AgenticComm} supports five query types:
\begin{enumerate}[nosep, leftmargin=*]
  \item \textbf{Search}: regex pattern matching across message content.
  \item \textbf{Semantic search}: cosine similarity over stored embedding vectors.
  \item \textbf{Timeline}: chronologically ordered messages within a time window.
  \item \textbf{History}: per-channel message retrieval with pagination.
  \item \textbf{Trust lookup}: O(1) HashMap access for agent trust levels.
\end{enumerate}

Trust lookup is critical for real-time message routing and achieves constant-time performance regardless of agent population (33.2\,ns at 100 agents, 34.7\,ns at 10K---Section~\ref{sec:query_perf}).

\subsection{Message Processing Pipeline}
\label{sec:pipeline}

The \textsc{MessageEngine} provides a unified processing pipeline for all message operations.
For send operations, the pipeline executes four stages: (1)~trust validation---verify the sender meets the channel's minimum trust level; (2)~consent check---verify the sender has an active consent grant for the channel; (3)~rate limiting---enforce per-agent message rate limits; (4)~routing and persistence---deliver the message and flush to the \texttt{.acomm} file.

For query operations, the engine dispatches to the appropriate query type (search, semantic, timeline, history) with unified error handling and audit logging.

\textbf{Encryption.}
Messages can be encrypted with ChaCha20-Poly1305 or AES-256-GCM before storage.
Both algorithms achieve sub-microsecond encryption latency (895\,ns and 949\,ns respectively on our benchmark hardware).
Key management integrates with \textsc{AgenticIdentity}~\cite{agenticidentity2026} for Ed25519-based key exchange.

\textbf{Hive Minds.}
For collaborative multi-agent workflows, \textsc{AgenticComm} supports hive mind formation: a group of agents that share a coherence threshold, a consensus policy, and optionally a unified cognitive space.
Hive formation and join operations scale linearly---23.4\,$\mu$s for 100 participants.

\section{Evaluation}
\label{sec:evaluation}

\subsection{Benchmark Setup}

All measurements were taken on a single machine:
\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{CPU}: Apple M4 Pro (ARM64)
  \item \textbf{RAM}: 64\,GB unified memory
  \item \textbf{OS}: Darwin 25.3.0
  \item \textbf{Compiler}: Rust 1.90.0 (release mode, LTO)
  \item \textbf{Framework}: Criterion 0.5 with 100-sample minimum
  \item \textbf{Date}: 2026-02-28
\end{itemize}

We benchmark five categories at three scales (100, 1K, 10K entities): file I/O (create, save, load), entity operations (message send, channel create, hive form), core computations (topic match, trust lookup, consent check, encryption), queries (search, semantic search, timeline, history), and the end-to-end write engine pipeline.
Rate limits are disabled for benchmark throughput (\texttt{messages\_per\_minute = u32::MAX}).

\section{MCP Interface Design}
\label{sec:mcp}

\textsc{AgenticComm} exposes its functionality through 17 MCP tools, consolidated from an initial design of 204 individual tools using an \emph{operation parameter pattern}.
Each tool accepts an \texttt{operation} string parameter that selects the specific action:

\begin{lstlisting}[language={}]
{
  "name": "comm_message",
  "operation": "send",
  "channel_id": "ch_001",
  "content": "Build complete",
  "message_type": "notification"
}
\end{lstlisting}

This consolidation reduces the MCP tool registry from $\sim$20,400 tokens (204 tools $\times$ $\sim$100 tokens each) to $\sim$1,700 tokens (17 tools), a 12$\times$ reduction in context window cost for AI assistants.
The 17 tools cover: \texttt{comm\_channel}, \texttt{comm\_message}, \texttt{comm\_semantic}, \texttt{comm\_affect}, \texttt{comm\_hive}, \texttt{comm\_consent}, \texttt{comm\_trust}, \texttt{comm\_keys}, \texttt{comm\_federation}, \texttt{comm\_temporal}, \texttt{comm\_query}, \texttt{comm\_forensics}, \texttt{comm\_collaboration}, \texttt{comm\_workspace}, \texttt{comm\_session}, \texttt{comm\_agent}, and \texttt{comm\_store}.

Error handling uses two-tier signaling: protocol errors remain JSON-RPC errors, while domain errors return structured tool responses with explicit failure state and the \texttt{isError: true} flag.
Unknown tools return error code $-32803$ (\texttt{TOOL\_NOT\_FOUND}).

\section{Discussion}
\label{sec:discussion}

\subsection{What This Enables}

\textsc{AgenticComm} enables five capabilities not available in current multi-agent frameworks:
(1)~\textbf{Auditable agent conversations}: the \texttt{.acomm} file provides a complete, tamper-evident record of all inter-agent communication, enabling post-hoc analysis of decision chains;
(2)~\textbf{Trust-gated collaboration}: channels can require minimum trust levels computed from interaction history, preventing untrusted agents from participating in sensitive workflows;
(3)~\textbf{Portable communication history}: a single file can be copied, backed up, or shared without infrastructure dependencies;
(4)~\textbf{Semantic search over conversations}: stored embedding vectors enable retrieval-augmented generation over historical agent interactions;
(5)~\textbf{Hive mind coordination}: structured group consensus protocols enable collective decision-making with configurable coherence thresholds.

\subsection{Limitations}

\textbf{Semantic search scaling.}
The brute-force cosine similarity search reaches 7.17\,ms at 10K messages.
For stores exceeding 100K messages, approximate nearest-neighbor indexing (HNSW or IVF) would be required.
This is a known limitation of the current implementation, not the architecture---the embedding storage format supports index overlay.

\textbf{Single-process model.}
\textsc{AgenticComm} operates within a single process.
Cross-process and cross-machine agent communication requires the federation extension (implemented but not benchmarked here).
The local-first model is deliberate: it eliminates network failure modes for the common case of co-located agents.

\textbf{File size limit.}
The 2\,GB file size limit (imposed by the 96-byte header's offset fields) supports $\sim$29 million messages.
Workloads exceeding this require file rotation, which is supported by the workspace abstraction but adds operational complexity.

\subsection{Future Work}

Three extensions are planned:
(1)~networked federation with signed message relay across trust zones;
(2)~hierarchical channel composition for organizational modeling;
(3)~integration with \textsc{AgenticContract}~\cite{agenticcontract2026} for enforceable communication SLAs.

\section{Conclusion}
\label{sec:conclusion}

We have presented \textsc{AgenticComm}, a structured communication library for multi-agent AI systems that stores typed, trust-aware messages in a compact binary format.
Our measured results demonstrate:
sub-microsecond message send (0.89\,$\mu$s/msg amortized),
constant-time trust lookup (34.7\,ns at 10K agents),
efficient persistence (9.71\,ms save, 8.67\,ms load for 10K messages),
and compact storage (71.8 bytes/message under Zstd compression).

\textsc{AgenticComm} addresses the communication gap in current agent frameworks by treating inter-agent messaging as a structured, persistent, identity-aware artifact rather than ephemeral string passing.
The system is implemented in Rust, exposed through 17 MCP tools, and designed for the 20-year timescale of agent communication history.
Source code is available at \url{https://github.com/agentralabs/agentic-comm}.
